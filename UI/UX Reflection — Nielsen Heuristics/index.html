Nielsen Heuristics Reflection

This is a reflection on the application of Nielsen's heuristics in UI/UX design.

Norman + Nielsen Reflection The heuristic that feels most aligned with Norman’s concerns in Chapter 1 is Visibility of System Status. This principle says that a system should always keep users informed about what is going on through clear and timely feedback. That idea connects directly to Norman’s focus on feedback and communication between the system and the user. In Chapter 1, he talks about how people get frustrated when objects do not clearly communicate what is happening or what action to take. Both authors are basically saying the same thing: when a system stays silent, users feel confused, and confusion leads to mistakes. Another reason this heuristic feels so connected to Norman is that both readings focus on reducing uncertainty. He uses everyday objects like doors and faucets to show how confusing design forces people into trial and error. Nielsen translates that same concern into interface design by saying users should always know what is happening inside the system. In both cases, the goal is to make interactions intuitive. 1. Visibility of System Status One heuristic my micro-project struggles with is visibility of system status. This principle is about making sure the user always knows what is happening inside the system. In my project, the system changes internally over time, especially when the user is idle or fails multiple verification attempts, but those changes are not always visible to the user. This principle relates to providing transparency so that the user has an accurate representation of what is occurring within the system. In my example project, there may be a lot of internal changes taking place while the user is not active on the system (or has not successfully verified themselves numerous times). However, the changes that occur are not always visible to the user. The issue of suspicion increases during periods of inactivity. The system tracks time and applies pressure incrementally in the background. However, nothing displays on the user's screen to signal that the state or the internal rules of the system have changed. The user may believe that the system is still functioning in a stable state when, in fact, the rules have changed, the system is more likely to deny access, and there is no timer, warning, or progress indicator displayed. This matters because users rely on feedback to build trust and understanding. When the system changes without communicating those changes, the user can feel confused or unfairly treated. From a design perspective, it creates a gap between what the system knows and what the user knows. In my project, that gap is partly intentional because I want the system to feel opaque and controlling, but it still shows a tension between usability and concept. A well-designed system usually keeps the user informed, even if the outcome is negative. 2. Match Between System and the Real World Another heuristic my micro-project struggles with is the match between the system and the real world. This principle means that systems should behave in ways that feel logical and familiar based on real-life expectations. In my project, the relationship between user action and system outcome is not always predictable, and sometimes the system denies access even when the user follows the instructions correctly. This issue occurs during the verification process. The user interacts with the interface by attempting verification, but success is determined by hidden variables instead of performance. The system may reject the user without giving a concrete reason. In real-world situations, people expect a clear cause-and-effect relationship. If they do the right thing, they expect a positive result. My system breaks that expectation on purpose.
