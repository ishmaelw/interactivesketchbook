Interface Audit — Micro-Project Page

Using Norman and Nielsen as Analytical Tools

This audit focuses on how my interface communicates structure, rules, and interaction. The goal is to see how it mediates the relationship between the user and the system. My micro-project centers on a verification system that escalates pressure over time, so the interface has to communicate changing states, restricted access, and interaction clearly. Right now, the system logic is strong, but the communication layer is still developing.

Visibility of System Status

The page partially communicates where the user is, but not consistently. When the system begins, the user sees a prompt to verify, and after interaction, they receive messages like “verifying,” “access denied,” or “access granted.” These messages provide basic feedback and indicate that the system has responded. The user is not always aware of what stage they are currently in, especially as the system escalates suspicion or difficulty.

Its not fully clear what state the system is in at any given moment. For example, the system increases pressure when the user is idle, but there is no indicator showing that escalation is happening. The user might believe nothing has changed, even though the internal rules have shifted.

Its obvious that clicking or pressing is interactive, because the system responds immediately. However, the overall system status remains partially hidden.

Weakness

The interface does not clearly communicate state progression over time. The user cannot easily tell whether they are in an early stage, a warning stage, or a restricted stage.

Affordances & Signifiers

The primary action available on this page is clicking or pressing to attempt verification. That is the core affordance of the system. The signifier is the instruction text telling the user to verify or interact. The changing messages after each attempt also signal that interaction is expected.

However, some actions are unclear or hidden. The system tracks idle time as a variable, but there is no visible signifier showing that inactivity affects the outcome. This creates a situation where the system reacts to behavior the user does not realize is being monitored.

Nothing appears clickable in a traditional sense, because the grid was removed and replaced with a pixel background. This makes the interaction more abstract. The user can interact anywhere, but the interface does not clearly show where interaction should occur.

Precision

  • Action available: Click or press to verify
  • Signifier: Instruction text and system messages
  • Unclear interaction: Idle time affecting suspicion
  • Hidden affordance: Time itself is an input variable

Navigation & Mapping

Navigation on this page is minimal because the interaction is contained within a single screen. The user does not move between pages but instead moves between system states. The layout reflects the system structure in a simple way:

  • Instruction at the top
  • Interaction area in the center
  • Feedback messages appearing during interaction

Hierarchy is somewhat clear, but not strongly reinforced. The user can identify the main instruction, but secondary information about system behavior is not visible. The mapping between input and outcome is partially predictable. Clicking leads to a response, but the result is not always consistent.

A user might hesitate because they are unsure how success is determined. They may also hesitate after repeated failures, wondering whether they are doing something wrong or whether the system has changed.

Hesitation Point

After multiple failed attempts, the user may not know whether to:

  • Keep trying
  • Wait
  • Refresh the page

Constraints & Assumptions

The page assumes the user understands that verification requires repeated attempts. It also assumes the user will recognize that time and persistence affect the system, even though these rules are not explicitly stated.

Several constraints exist in the system. The user:

  • Cannot bypass the verification process
  • Cannot see the exact requirements for success
  • Cannot reset the system state manually

These restrictions limit user control and reinforce the system’s authority.

Hidden elements include:

  • The suspicion variable
  • Escalation thresholds

Identified Assumption

The interface assumes the user understands that the system is dynamic and changing over time, even though that change is not clearly communicated.

Heuristic Violations

1. Visibility of System Status

Violation: The system changes internally without clearly communicating those changes to the user.

Where it occurs: When suspicion increases during idle time or after repeated failures, there is no persistent indicator showing the escalation.

Why it matters: Users cannot build a reliable mental model of the system if they do not know what state it is in.

Structural Fix: Add a persistent stage or status indicator that updates as the system escalates, such as displaying the current verification level or system condition.

2. Match Between System and the Real World

Violation: User actions do not consistently lead to predictable outcomes.

Where it occurs: During verification attempts, the system may deny access even when the user follows instructions.

Why it matters: This breaks the expected cause-and-effect relationship between action and result, making the system feel unreliable.

Structural Fix: Provide a clear explanation or rule cue that indicates why failure occurs, such as a message showing increased restriction or system sensitivity.

Structural Revision Proposal

1. Strengthen Hierarchy

Introduce a persistent system status region that clearly separates instruction, interaction, and feedback. This would help users quickly understand the structure of the page and recognize the system’s current condition.

2. Clarify Navigation Through State Indicators

Add a visible progression marker that shows movement between states, such as:

  • Initial verification
  • Warning
  • Restriction
  • Access granted

3. Improve Interaction Clarity

Explicitly signal that inactivity affects the system. This could be done by displaying a countdown, timer, or warning message that appears before escalation occurs. This change would make hidden variables visible and reduce user confusion.

Bottom Line

My system logic is intentional and structured, but the interface still hides too much information from the user. The next phase of development should focus on making system states, rules, and consequences more legible so users can understand how their behavior connects to the system’s response.
