Shneiderman’s 8 Golden Rules — Reflection

1. Which rule feels most absent from your micro-project page?
The rule that feels most absent from my micro-project page is “Offer informative feedback.” My system creates pressure by escalating restrictions over time, but it does not always clearly communicate what is happening to the user or why certain actions are blocked. For example, when the verification step fails, the interface does not provide enough explanation about what changed in the system. The user is left to guess the reason behind the restriction. This absence makes the interaction feel impossible. If the system provided clearer feedback, the tension would still exist, but the user would better understand the logic behind the experience.

2. Which rule would most improve your sketchbook interface?
The rule that would most improve my sketchbook interface is “Reduce short-term memory load.” Right now, the user has to remember previous steps, rules, or states in order to continue interacting with the system. Information is not always visible on the page, so the user must rely on memory instead of the interface itself. Adding visible reminders—such as status indicators, progress markers, or instructions—would make the interaction smoother and more accessible. It would shift the responsibility from the user’s memory to the design of the system, which is one of the main goals of good interface design.

3. Which rule is hardest to implement honestly?
The hardest rule to implement is “Permit easy reversal of actions.” My system is intentionally designed to create pressure and consequences, so allowing users to easily undo actions would weaken that structure. The tension of the experience depends on the idea that certain decisions feel permanent. Implementing an undo function would contradict the emotional and conceptual goal of the project.

4. Did any rule contradict how you designed your system originally?
Yeah, the rule “Strive for consistency” partially contradicts how I originally designed my system. In my design, the interface changes as pressure increases, and those changes are meant to feel unstable or unpredictable. Pixels might move, access may shift, or rules may tighten over time. This inconsistency is intentional because it reflects the idea of escalating control. However, from a usability perspective, consistency usually helps users feel confident and oriented. This creates a tension between conceptual design and usability standards. Instead of completely rejecting consistency, I am learning to use controlled inconsistency as a deliberate design choice rather than an accident.
