“Premium Required”

Issue / behavior / system you want to align with: Digital paywalls and systems that gate basic functionality behind payment. Interaction rule (conceptual): The user can begin interacting freely at first, but once they reach a certain threshold of use, access becomes restricted. Actions that previously worked now trigger a “premium required” state. The restriction may appear arbitrary or timed rather than clearly rule-based. Input: Repeated clicks and scrolling attempts to unlock content. What changes over time: The number of allowed actions decreases; locked elements increase; access shrinks. Where friction appears: limits scrolling or partial visibility of content. Intended feeling: Exclusion and frustration.

"Verify You’re Human”

Issue / behavior / system you want to align with: Automated systems that force people to repeatedly prove authentication in order to access basic digital space. Interaction rule (conceptual): The user must complete a verification task before proceeding. Each time they successfully complete it, a new verification appears. The criteria subtly change or become slightly more ambiguous over time, even though the task looks similar. Input: Clicking, selecting items, or completing small challenges. What changes over time: The number of verifications increases; instructions become less clear; tolerance for “correct” responses shrinks. Where friction appears: Repetition, unclear rules, near-success followed by failure, the sense of being evaluated by something invisible. Intended feeling: Suspicion and humiliation.

“Are You Still There?”

Issue / behavior / system you want to align with: Attention-tracking systems that monitor inactivity and assume disengagement if you pause. Interaction rule (conceptual): The system constantly monitors user activity. If there is even a brief pause, it triggers a warning asking if the user is still present. Over time, the allowed inactivity window shrinks, requiring more frequent interaction to avoid being “timed out.” Input: Mouse movement or key presses. What changes over time: The tolerance for stillness decreases; warnings appear faster; timeouts become more aggressive. Where friction appears: Punishment for stillness, forced micro-movements, constant monitoring. Intended feeling: Anxiety and surveillance.

Micro-Project — Progress #1

Verify You’re Human

1. Concept Summary

This interaction simulates a system that forces certain users to complete a human verification task before accessing content. The user can click to attempt verification. A hidden system variable determines whether they are flagged for verification or allowed to pass instantly. Over time, repeated attempts increase suspicion, making it harder to exit the verification state.

2. Primary Variable

let flagged = false;

This variable determines whether the user is required to complete verification. It changes at the start of the sketch using a hidden rule (random assignment simulating a “risk score”). Visually and behaviorally, it controls whether the user immediately sees content (State 1) or is placed into a verification screen (State 2).

3. Base Build (Week 1 Implementation)

The first working version will include:

  • One variable: flagged
  • One input: Mouse click
  • Two visible states

State 1: Access Granted
The screen displays “Access Granted.” No verification required.

State 2: Verification Required
The screen displays “Please verify you are human.” Clicking attempts verification but does not guarantee exit.

4. Structural Growth

Over the next three weeks, this system can expand by adding escalation and instability. A stage variable could track how many verification attempts have occurred and increase task difficulty over time. Time-based decay could re-trigger verification even after successful completion. Friction could be introduced by shrinking response windows or increasing ambiguity. The system could also introduce uneven reset behavior, where some users are pushed back into verification more frequently than others.

Micro Project — Progress #2

The current build generates a randomized starting stage that determines whether the user must complete verification or is granted immediate access. If verification is required, a grid appears (either 3x3 or 4x4 depending on stage), and the user can click squares to select them. When all required squares are correctly selected, the system reduces the stage by one and generates a new grid until access is granted.

The main variable is stage, which controls the user’s verification level and grid size. When the user interacts by clicking squares and successfully matching the hidden correct pattern, the stage decreases, changing the grid size or eventually removing the verification entirely.

Additional variables track user's behavior, such as time spent in each stage, to further refine the verification process.
