Micro Project Self Assessment

Project Evaluation

Overall, I feel like this project is pretty strong. It started as a basic interaction using a grid-based CAPTCHA system, and then developed into a system that focuses more on algorithmic judgment and opacity. The shift from a solvable task to a system that feels punitive and impossible shows intentional development.

Core Rule & System Clarity (18 / 20)

The system has a clear rule: users must attempt verification repeatedly, but success is determined by an opaque system that can escalate or restrict access based on hidden conditions. This rule is consistent across the interaction and is reflected in both the code and the behavior of the system. There was a small inconsistency early on between variables, but it was resolved in later versions.

State Progression (14 / 15)

The system moves through clearly defined states, including start, verification, failure, idle escalation, and access granted. Each state changes how the system behaves rather than just how it looks. The progression between these states is meaningful and contributes to the overall structure.

Interaction & Evolution (14 / 15)

User interaction produces real consequences. Clicking attempts verification, inactivity increases suspicion, and failure escalates difficulty. The system evolves over time through increasing restriction and instability. The interaction is slightly abstract since clicking is not tied to a visible task anymore, but it still functions clearly within the system.

Iterative Development (15 / 15)

There is clear evidence of iteration across versions. The project began with a grid-based system, then introduced idle suspicion, probabilistic failure, and shrinking tolerance. The final version removes the grid entirely, shifting the focus away from user skill and toward system control. These changes show strong structural development.

Documentation & Logging (13 / 15)

The sketchbook documents the process with multiple versions, reflections, and explanations of rule changes. The evolution of the system is clear. However, it could be strengthened by including more detailed notes on code experiments or failed attempts, as well as a system diagram.

Risk & Experimentation (10 / 10)

The project pushes beyond a standard interaction. Removing the grid and relying on an opaque system was a strong conceptual decision. The introduction of bias, probabilistic failure, and hidden rules shows meaningful experimentation with system behavior.

Participation & Critique Engagement (9/ 10)

I participated in the group feedback in zoom and in breakout rooms which gave me clarity on what I needed to work on in my project. The project reflects ongoing adjustment and responsiveness to feedback, though more visible critique integration could strengthen this further.

Total Score: 93 / 100

I feel I demonstrated a strong understanding of system design and interaction.
